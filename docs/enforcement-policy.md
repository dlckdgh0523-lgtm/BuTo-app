# BUTO 제재 및 이의제기 정책

## 목적

BUTO의 제재 정책은 자동 영구정지를 기본값으로 두지 않습니다. 우선 목표는 위험 행위를 즉시 멈추고, 증거를 고정하고, 사용자가 이의제기를 제출할 수 있게 한 뒤, 운영팀이 최종 판정하는 것입니다.

## 핵심 원칙

1. 자동 제재의 기본값은 `즉시 잠금 -> 증거 고정 -> 운영 검토`입니다.
2. 사용자는 다른 테이블로 이동하지 않습니다. `users.status`만 변경합니다.
3. 제재 이력과 이의제기 이력은 별도 테이블에 누적 저장합니다.
4. 복구되어도 기존 제재 이력은 삭제하지 않습니다.
5. 자동 영구정지는 아주 좁은 범위의 중대 사유에만 허용합니다.

## 상태 모델

- `ACTIVE`: 정상 이용 가능
- `RESTRICTED`: 즉시 잠금 또는 제한 적용 상태. 운영 검토 또는 이의제기 가능
- `SUSPENDED`: 운영팀이 일정 기간 또는 정책 사유로 정지한 상태
- `APPEAL_PENDING`: 사용자가 이의제기를 제출했고 운영 판정이 진행 중인 상태
- `REINSTATED`: 제한 해제 후 재이용 허용 상태
- `PERMANENTLY_BANNED`: 영구 정지 상태
- `WITHDRAWN`: 사용자 요청에 의한 탈퇴 상태

## 기본 처리 흐름

1. 위험 신호 감지
AI 모더레이션, 결제 리스크, 위치 증빙 위반, 관리자 정책 판정 등으로 위험 신호를 감지합니다.

2. 즉시 잠금
자동 제재는 기본적으로 `RESTRICTED`를 적용합니다. 이 단계에서는 채팅, 매칭, 결제 등 고위험 기능을 실패-폐쇄 방식으로 막습니다.

3. 증거 고정
제재와 동시에 증거 번들을 생성합니다. 이 번들에는 원본 메시지 요약, 위치 증빙 요약, 운영 메모 등 판정 근거가 포함됩니다.

4. 이의제기
사용자는 제재 대상 action을 기준으로 이의제기를 제출할 수 있습니다. 제출 시 `users.status`는 `APPEAL_PENDING`으로 바뀌고, 원래 제재 이력은 유지됩니다.

5. 운영 판정
- 추가 자료 요청: `MORE_INFO_REQUESTED`
- 승인: 제재 해제, `users.status`를 `REINSTATED`로 변경
- 기각: 원래 제재 상태로 복귀 또는 유지

## 자동 영구정지 허용 범위

자동 영구정지는 운영·법무 합의가 끝난 극소수 사유만 허용합니다. 현재 코드 기준 허용 범위는 아래와 같습니다.

- `CONFIRMED_CREDENTIAL_THEFT`
- `CONFIRMED_PAYMENT_FRAUD`
- `CHILD_SAFETY_CRITICAL`

이 외의 자동 제재는 기본적으로 `RESTRICTED` 또는 `SUSPENDED`로 두고, 수동 판정을 거쳐야 합니다.

## 수동 판정 범위

아래 유형은 자동 영구정지로 처리하지 않습니다.

- 부적절 텍스트 또는 금칙어 탐지
- 위치 위조 의심
- 결제 이상 징후
- 신고 누적
- 후기 조작 또는 커뮤니티 남용

이 경우 운영팀은 증거와 이력을 확인한 뒤 `RESTRICTED`, `SUSPENDED`, `REINSTATED`, `PERMANENTLY_BANNED` 중 하나로 판정합니다.

## 정지 해제 운영 원칙

- 정지 해제는 사용자를 다른 보관 테이블로 옮기는 작업이 아닙니다.
- `users.status`를 `ACTIVE` 또는 `REINSTATED`로 변경합니다.
- 기존 `user_enforcement_actions` 레코드는 삭제하지 않습니다.
- 해제 시각과 해제 오퍼레이터 정보는 감사 로그와 제재 액션의 `liftedAt`, `liftedByActionId`로 남깁니다.

## 이력 저장 원칙

### users

- 단일 계정 레코드를 유지합니다.
- 현재 사용 가능 여부는 `users.status`가 단일 진실 원천입니다.

### user_enforcement_actions

- 제재의 누적 이력 테이블입니다.
- 어떤 사유로 어떤 범위의 제한을 적용했는지 보존합니다.
- 해제되더라도 삭제하지 않습니다.

### enforcement_evidence_bundles

- 제재 근거를 고정하는 증거 묶음입니다.
- 원본 메시지 전문 대신 최소 필요 요약과 운영 판정 근거를 저장합니다.

### user_appeals

- 사용자 이의제기 원문과 상태를 누적 저장합니다.
- 한 action에 동시 진행 중인 열린 appeal은 하나만 허용합니다.

### appeal_review_actions

- 운영팀의 이의제기 처리 이력을 누적 저장합니다.
- 추가 자료 요청, 승인, 기각 판정을 모두 남깁니다.

## 사용자 UX 원칙

정지 화면은 다음을 반드시 보여줍니다.

- 문구: `계정 이용이 일시 제한되었어요`
- 제재 사유 코드
- 제한 범위
- 검토 상태
- 이의제기 제출 버튼
- 제재 사유 보기 버튼
- 고객센터 문의 버튼

고객센터 문의는 별도 인앱 채팅이 준비되기 전까지 카카오톡 채널로 연결합니다.

## 운영 리스크 메모

- 자동 영구정지 범위는 출시 전 운영·법무 승인 없이는 넓히지 않습니다.
- 증거 번들은 사후 분쟁 재구성에 필요한 수준만 저장하고 과도한 개인정보는 피합니다.
- 정지 해제는 실수 한 번으로 전체 리스크가 열리므로 감사 로그가 필수입니다.
